# 多平台发布产品设计定稿（v3）

> 版本：v3 | 日期：2026-06-25 | 状态：定稿

---

## 一、核心理念

> **文章只有一个 ID，一份 contentJson。发布记录只记"发出去的那一刻"。**

---

## 二、数据模型（2 张表）

### 2.1 content_article — 文章主表

```
字段               类型          说明
─────────────────────────────────────────────────
id                 BIGINT PK     唯一文章 ID
title              VARCHAR(200)  文章标题
slug               VARCHAR(200)  URL 友好标识
summary            TEXT          摘要
cover_url          VARCHAR(500)  封面图
tags               VARCHAR(500)  标签
content_json       LONGTEXT      编辑器最新内容（唯一正文源，可随时修改）
content_html       LONGTEXT      渲染后的 HTML
content_md         LONGTEXT      Markdown 版本
style_preset       VARCHAR(100)  当前选中的样式预设
status             VARCHAR(20)   DRAFT / PUBLISHED
visibility         VARCHAR(20)   PUBLIC / PRIVATE
seo_title          VARCHAR(200)  SEO 标题
seo_description    VARCHAR(500)  SEO 描述
seo_keywords       VARCHAR(500)  SEO 关键词
canonical_url      VARCHAR(500)  规范 URL
author_id          BIGINT        作者 ID
author_name        VARCHAR(64)   作者名
created_by         VARCHAR(64)   创建人
created_at         DATETIME
updated_at         DATETIME
published_at       DATETIME
del_flag           CHAR(1)       '0' / '1'
```

### 2.2 article_publish — 发布记录

```
字段                    类型          说明
────────────────────────────────────────────────────────────
id                      BIGINT PK     主键
article_id              BIGINT        FK → content_article.id
platform                VARCHAR(50)   平台编码: WECHAT / SITE_WENDAO / ZHISHU_XINGQIU ...

平台特有信息:
platform_title          VARCHAR(200)  该平台标题（NULL=使用主标题）
platform_tags           VARCHAR(500)  该平台标签

发布状态:
status                  VARCHAR(20)   PENDING / PUBLISHING / SUCCESS / FAILED / DELETED

内容快照（仅用于历史追溯，不做逻辑判断）:
content_json_snapshot   LONGTEXT      该平台发出去时的 contentJson 快照

平台返回信息:
platform_article_id     VARCHAR(200)  平台上的文章 ID
platform_url            VARCHAR(500)  平台上的链接

错误与重试:
error_code              VARCHAR(100)  错误码
error_message           TEXT          错误描述
retry_count             INT DEFAULT 0 重试次数

有效标记:
is_active               BOOLEAN DEFAULT TRUE  FALSE=已从该平台删除

时间:
published_at            DATETIME      成功发布时间
created_at              DATETIME
updated_at              DATETIME

约束:
UNIQUE KEY (article_id, platform)
```

## 三、产品规则

### 规则 1：发布即定稿

```
文章发布到某平台成功 = 定稿。
CoAI 不支持通过 CoAI 修改已在外部平台成功发布的内容。

要改已发布的内容：
  · 去平台后台改（微信：仅文字，最多 3 次）
  · 或者删除后重新新建文章发布

例外：问道平台
  问道读取 CoAI 的 content_json（实时数据）
  修改 content_json → 问道自动同步
  不需要额外操作，不需要 [更新问道] 按钮
```

### 规则 2：失败可重试

```
某平台发布失败 → 运营可以：
  1. 在编辑器修改正文（修复失败原因）
  2. 点该平台的 [重试]
  3. 后端用当前最新的 content_json 重发
  4. 更新该平台的 content_json_snapshot

已成功的平台不受影响（它们的 snapshot 没变）。
```

### 规则 3：快照用于历史查看，不做逻辑判断

```
content_json_snapshot 仅用于"查看该平台当时发出去的是什么"。
不用于版本对比、不用于一致性检测、不用于弹窗警告。

各平台 snapshot 独立存储，互不影响。
```

### 规则 4：删除 = 标记

```
不物理删除 article_publish 记录。
设置 is_active = FALSE。
删除失败的平台可重试。
不支持的平台提示手动删除。
```

## 四、完整流程

### 4.1 新建 + 一键发布

```
1. 运营在编辑器写文章
2. 勾选 ☑问道 ☑微信 ☑星球 → 点 [发布]
3. 后端:
   → 创建 content_article (status=PUBLISHED)
   → 为每个平台创建 article_publish (status=PENDING)
   → content_json_snapshot = 当前 content_json
   → 异步队列处理各平台

结果示例:
  问道 ✅ → publish.status=SUCCESS
           (问道读实时数据，自动同步)
  微信 ⏳ → publish.status=DRAFT_CREATED
           (草稿已推送，需用户在微信后台发布)
  星球 ✅ → publish.status=SUCCESS

前端展示:
  ┌─ 文章详情 ──────────────────────────────┐
  │  🔹 格物科技深度解析                      │
  │                                          │
  │  发布状态:                                │
  │  问道 ✅ 已发布                           │
  │  微信 ⏳ 草稿已推送 [打开草稿] [已发布确认]  │
  │  星球 ✅ 已发布                           │
  └──────────────────────────────────────────┘
```

### 4.2 部分成功 / 部分失败

```
结果示例:
  问道 ✅ → status=SUCCESS
  微信 ❌ → status=FAILED, error_message="素材空间不足"
  星球 ✅ → status=SUCCESS

前端展示:
  问道 ✅ 已发布
  微信 ❌ 素材空间不足 [重试] [忽略]
  星球 ✅ 已发布

规则:
  · 成功的平台: 不碰
  · 失败的平台: 标记为失败，用户可 [重试] 或 [忽略]
  · 各平台独立，失败不影响其他平台
```

### 4.3 修改 + 重试失败平台

```
时间线:
  T0: content_json = v1
  T1: 发布 → 问道 ✅ (snapshot=v1) 星球 ✅ (snapshot=v1) 微信 ❌ (snapshot=v1)
  T2: 运营在编辑器修改正文 → content_json = v2
  T3: 运营点微信 [重试]

后端:
  → 微信.snapshot = 当前 content_json (v2)
  → 微信.status = PENDING → 入队 → SUCCESS ✅
  → 问道.snapshot 不碰（还是 v1）
  → 星球.snapshot 不碰（还是 v1）

结果:
  content_json = v2（编辑器最新内容）
  问道: 实时读到 v2 → 自动同步
  微信: 已发布, snapshot=v2
  星球: 已发布, snapshot=v1（发出去时就是 v1）

  各平台 snapshot 独立，互不影响。
```

### 4.4 查看各平台历史版本

```
文章详情 → 点某平台发布卡片:

┌─ 知识星球 ─────────────────────────────────┐
│  ✅ 已发布 (06-25 10:25)                    │
│  链接: zsxq.com/article/xxx                 │
│                                             │
│  [查看发往星球的版本]                         │
│                                             │
│  点开后 → 用 publish.content_json_snapshot   │
│  渲染预览，展示当时发出去的内容               │
└────────────────────────────────────────────┘

各平台预览数据来源:
  问道: 直接从 content_article 读（实时最新，不需要 snapshot）
  微信: publish.content_json_snapshot（历史快照）
  星球: publish.content_json_snapshot（历史快照）
  知乎: publish.content_json_snapshot（历史快照）
```

### 4.5 删除

```
运营点 [删除文章] → 弹窗:

  ☑ 同时从平台删除（调用各平台删除 API）
  ☐ 仅在 CoAI 删除，保留已发布内容

后端:
  for each publish where status=SUCCESS:
    平台支持删除 API → 调 API → is_active=FALSE
    平台不支持删除  → 提示手动删除 → is_active=FALSE
  content_article 标记删除 (del_flag='1')
```

## 五、关于问道的特殊说明

```
问道的文章展示接口直接读取 CoAI 的 content_json（实时数据）。

这意味着：
  修改 CoAI 的 contentJson → 问道自动同步最新内容
  不需要 [更新问道] 按钮
  不需要 content_json_snapshot（问道不需要记录历史版本）
  snapshot 只给外部平台（微信/知乎/星球等）使用

这是问道作为自有平台的优势，其他外部平台不具备。
```

## 六、不需要的功能

```
以下功能在 v3 设计中已明确移除：

❌ 版本管理表（article_version）
   原因: 不需要记录每次正文修改的历史版本

❌ 版本号（published_version）
   原因: 一个 snapshot 字段即可记录最后一次发布的内容

❌ 快照对比逻辑
   原因: snapshot 仅用于展示，不做逻辑判断

❌ 一致性警告弹窗
   原因: 编辑器最新内容 ≠ 已发布内容，这是正常现象，不是错误

❌ 限制重试次数
   原因: 发布失败重试 和 已发布修改 是两回事。
         我们的设计不支持修改已发布内容，所以不存在"3 次限制"的问题。
         发布失败的重试没有次数限制的必要。
```

## 七、关于设计过程中的疑问回复

### 问题 1：为什么不用多 article_id？每个平台独立 ID 不是更清晰吗？

```
如果每个平台都用一个独立 article_id：
  · 列表里一条内容出现 N 条记录，运营要自己关联"它们其实是同一篇"
  · 改了错别字要改 N 个地方的 contentJson
  · 问道需要额外关联逻辑才能知道"这些是同一篇的不同版本"

而单 ID + publish 记录的设计：
  · 一条文章，各平台发布状态在下面展开，一目了然
  · 改正文只需要改 content_article 一处
  · 问道读到的是实时数据，不需要额外逻辑

结论：单 ID 更适合"内容同源、多平台分发"的场景。
竞品（MultiPost、TurboPush、Buffer 等）均采用单 ID 策略。
```

### 问题 2：改了 contentJson 重试微信后，问道读到的是新内容，那"已发布版本"不准了怎么办？

```
content_article 的 contentJson 代表"编辑器当前最新版本"。
article_publish 的 content_json_snapshot 代表"发出去时的版本"。

这是两个不同的概念。

问道读的是 contentJson（实时数据）—— 所以它自动变成新内容。
这没有问题。问道上的内容就是 CoAI 当前的内容。
不需要旧版本记录，因为问道和 CoAI 是同一套系统。

微信的 snapshot 是 v2（重试时记录的）。
微信实际发布的内容和 snapshot 一致。
这个准不准？准。

星球的 snapshot 是 v1（首次发布时记录的）。
星球实际发布的内容和 snapshot 一致。
这个准不准？准。

所有"不准"的担忧，来自于把"编辑器最新内容"和"已发布内容"混淆了。
它们本来就应该是两个东西。
```

### 问题 3：某平台重试失败 3 次了，要不要限制？

```
不需要。

发布失败重试 和 已发布文章的修改 是两回事：
  - 已发布文章的修改 → 微信只支持 3 次文字修改
  - 发布失败的重试 → 文章还没成功发出去，只是重新提交

失败重试的原因很多（网络波动、素材上传超时、平台临时故障），
限制次数反而让运营无法操作。

我们的设计：不限制重试次数，运营想重试几次就重试几次。
```

### 问题 4：如果运营在平台后台手动修改了已发布的内容，CoAI 怎么同步？

```
同步不了。也不需要同步。

article_publish 的语义是：
  "CoAI 在某个时刻成功将内容推送到了该平台"
  不是 "该平台上现在的内容永远和 CoAI 一致"

所有多平台发布工具都是这样的。
Buffer 不知道用户在 Twitter 后台改了文案。
易媒助手不知道用户在抖音换了封面。

这是系统边界的问题，不是设计缺陷。
```

### 问题 5：竞品都是怎么处理"部分成功部分失败"的？

```
竞品采用和本设计完全一致的策略：
  · 一条文章，各平台独立发布
  · 成功的标记成功，失败的标记失败
  · 失败的可重试（原内容）
  · 不影响已成功的平台

竞品和本设计的差异：
  竞品：不支持失败后修改正文再重试（想改就新建文章）
  本设计：支持失败后修改正文再重试

竞品不做这个功能的原因：
  竞品面向的是"内容一模一样分发到各平台"的场景
  本设计面向的是"各平台内容可以有差异"的场景
  业务需求不同，设计取舍不同。
```
